Topic 01

Disk Geometry

Physical Components
ComponentDefinition
PlatterRigid circular disk coated with magnetic material. Each platter has 2 surfaces.
TrackConcentric circular path on a surface where data is magnetically recorded.
SectorSmallest storage unit. Typically 512 bytes or 4096 bytes.
CylinderAll tracks at the same arm position across ALL platter surfaces (stacked). No arm movement needed to access a full cylinder.
R/W HeadFlies just above each surface. One per surface; all attached to one actuator arm.
Access Time Components
Access Time = Seek Time + Rotational Latency + Transfer Time
  • Seek Time — arm moves to correct track (ms)
  • Rotational Latency — wait for sector to rotate under head
  • Transfer Time — actual data read/write
Avg Rot. Latency = ½ × (60/RPM × 1000) ms
Worst-case = 1 × (60/RPM × 1000) ms
Capacity = Platters × 2 × Tracks/Surface × Sectors/Track × Bytes/Sector
Worked Problems
Q1. Disk at 7200 RPM — find time per rotation & avg rotational latency ▼

Time per rotation = 60 / 7200 = 0.00833 s = 8.33 ms

Avg rotational latency = 8.33 / 2 = 4.17 ms

Q2. Seek=9ms, 6000 RPM, Transfer=0.6ms — total access time ▼

Time per rotation = 60/6000 = 10 ms → Avg rotational latency = 5 ms

Access Time = 9 + 5 + 0.6 = 14.6 ms

Q3. 5 platters, 2 surfaces, 1500 tracks, 400 sectors, 512 B/sector — capacity ▼

Total surfaces = 5 × 2 = 10

Total sectors = 10 × 1500 × 400 = 6,000,000

Total bytes = 6,000,000 × 512 = 3,072,000,000 bytes

Convert: 3,072,000,000 / 1,048,576 ≈ 2930 MB ≈ 2.93 GB

Q4. Seek=8ms, 5400 RPM, Transfer=0.5ms — total access time ▼

60/5400 = 11.11 ms → Avg latency = 5.55 ms

Access Time = 8 + 5.55 + 0.5 = 14.05 ms

Q5. Seek=7ms, 7200 RPM, Transfer=0.4ms — WORST-CASE (sector just missed) ▼
When the request arrives just AFTER the desired sector has passed, you must wait a FULL rotation — not half.

Time per rotation = 60/7200 = 8.33 ms

Worst-case rotational delay = full rotation = 8.33 ms

Worst-case Access Time = 7 + 8.33 + 0.4 = 15.73 ms

Memory Virtualization

Address Space

Set of all memory addresses a program can generate. Each process sees its own private address space.

TypeDescription
Logical/VirtualGenerated by CPU. Starts at 0. What the program sees.
PhysicalActual RAM location. Managed by OS + hardware.
Process Layout (low → high)
High ↑ Stack (grows ↓) — local vars, return addresses │ │ Heap (grows ↑) — malloc/dynamic alloc │ │ BSS — uninit globals (zeroed at runtime) │ Data — init globals (int g = 10) Low ↓ Text — compiled code (read-only)
Base & Bounds

Each process gets two CPU registers:

  • Base — starting physical address

Physical Address = Base + Logical Address
If Logical Address ≥ Bound → TRAP

Topic 04

Paging

Core Concept

Virtual memory → pages (fixed size). Physical memory → frames (same size). Any page maps to any frame — no contiguous allocation needed.

Page Number = VA ÷ Page Size (integer)
Offset = VA mod Page Size
PA = (Frame Number × Page Size) + Offset

VA structure: If VA space = 2m, page size = 2n:

VA = [ Page# (m-n bits) | Offset (n bits) ] Offset bits = log₂(Page Size) Page# bits = m - n
Eliminates external fragmentation. May have internal fragmentation (last page not fully used).
Page Table Entry (PTE)
FieldMeaning
Frame NumberPhysical frame where page is stored
Valid Bit1=in memory, 0=not present (page fault)
Dirty BitModified since loaded — must write back
Reference BitRecently accessed — used in replacement
ProtectionR/W/X permissions

Page Fault steps:

Hardware detects invalid bit → page fault exception → OS takes control

OS checks if access is valid (belongs to process?)

Locate page on disk (swap space)

Load page into free frame (run replacement if no free frame)

Update PTE: set frame#, valid=1

Restart the faulting instruction

Worked Problems
Q1. Page=1KB, VA=2500, page 2→frame 7 — find PA▼

Page# = 2500 ÷ 1024 = 2, Offset = 2500 mod 1024 = 452

PA = (7 × 1024) + 452 = 7168 + 452 = 7620

Q2. Page=2KB, VA=6000, page 2 invalid — page fault?▼

Page# = 6000 ÷ 2048 = 2 → Page 2 is invalid → PAGE FAULT occurs

Q3. Page=2KB, VA=0x5A3C7 — page# and offset▼

0x5A3C7 → (5×65536)+(10×4096)+(3×256)+(12×16)+7 = 327680+40960+768+192+7 = 369,607

Page# = 369,607 ÷ 2048 = 180

Offset = 369,607 mod 2048 = 967

Q4. VA space=64MB, page=4KB — pages, bits▼

64MB = 2²⁶, 4KB = 2¹²

Offset bits = 12 | Pages = 2²⁶/2¹² = 2¹⁴ = 16,384 | Page# bits = 14 | Total LA bits = 26

Q5. VA=30 bits, page=2KB, PTE=4 bytes — #pages and page table size▼

VA space = 2³⁰, page = 2¹¹ → pages = 2³⁰/2¹¹ = 2¹⁹ = 524,288

Page table size = 524,288 × 4 = 2 MB

Practice: 64 pages of 1024 words, 32 frames — bits in LA and PA▼

(a) LA bits: 64 pages = 2⁶ → 6 page bits + 1024 words = 2¹⁰ → 10 offset bits = 16 bits

(b) PA bits: 32 frames = 2⁵ → 5 frame bits + 10 offset bits = 15 bits

Topic 05

TLB — Translation Lookaside Buffer

What is TLB?

Without TLB: every memory access needs 2 memory accesses (page table + data). TLB caches recent page→frame mappings to reduce this to ~1 access.

TLB is associative memory — ALL entries searched in parallel (not sequential). Part of the CPU instruction pipeline = near-zero overhead on hit.

PropertyValue
Size32 to 1,024 entries (small)
SpeedPart of instruction pipeline
Hit ratio90–99% in modern systems
ASIDTags entries per-process (avoids full flush on context switch)
TLB Hit vs Miss
EventMemory AccessesWhat happens
HIT 1 Page# found in TLB → frame# immediately → access data
MISS 2 Page# not in TLB → access page table in RAM → get frame# → access data. Entry loaded into TLB.
Hit Ratio = TLB Hits / Total Memory Accesses
Context Switch + No ASID: TLB must be fully flushed — all entries invalidated — otherwise process A could use process B's mappings. With ASIDs, multiple processes can coexist in TLB.
Topic 06

EMAT — Effective Memory Access Time

Formulas

Without TLB overhead:

EMAT = (α × mem) + ((1-α) × 2×mem)

Where α = hit ratio, mem = memory access time

With TLB overhead (h ns for lookup):

Hit time = h + mem
Miss time = h + 2×mem
EMAT = α(h+mem) + (1-α)(h+2×mem)
Worked Problems
Q1. mem=100ns, no overhead. Hit=80% and Hit=99%▼

Hit=80%: EMAT = 0.80×100 + 0.20×200 = 80+40 = 120 ns

Hit=99%: EMAT = 0.99×100 + 0.01×200 = 99+2 = 101 ns

Q2. mem=200ns, TLB overhead=20ns, hit=90% (PDF main example)▼

(a) Without TLB: 2 × 200 = 400 ns

(b) Hit time = 20 + 200 = 220 ns

Miss time = 20 + 200 + 200 = 420 ns

EMAT = (0.9 × 220) + (0.1 × 420) = 198 + 42 = 240 ns

Q3. mem=50ns, hit=75%, TLB lookup=2ns (textbook)▼

(a) No TLB: 2 × 50 = 100 ns

(b) Hit time = 2+50 = 52 ns. Miss time = 2+50+50 = 102 ns.

EMAT = 0.75×52 + 0.25×102 = 39 + 25.5 = 64.5 ns

Topic 07

Multi-Level Paging

The Problem

32-bit VA space + 4KB pages:

Pages = 2³² / 2¹² = 2²⁰ = 1,048,576 At 4 bytes/PTE → 4 MB per process 100 processes → 400 MB just for page tables!

Worse: most of the virtual address space is sparse (unused) but flat table allocates entries for ALL pages.

Single-level page table must also be stored contiguously in RAM.
The Solution

Page the page table itself. Only create inner tables for virtual regions actually in use.

VA = [ L1 Index | L2 Index | Offset ]

L2 bits = Total VA bits − L1 bits − Offset bits
L1 entries = 2^(L1 bits)
L2 entries = 2^(L2 bits)
Sparse process: most inner page tables never created → saves huge amounts of memory.
Worked Problems
Q1. VA=36 bits, page=8KB, L1 index=10 bits — find L2 bits, entries▼

Offset bits = log₂(8KB) = log₂(8192) = 13

L2 bits = 36 − 10 − 13 = 13 bits

L1 entries = 2¹⁰ = 1024

L2 entries = 2¹³ = 8192

Textbook: 32-bit, 1KB page — two-level structure▼

Offset = 10 bits. Remaining = 22 bits → split as 12 + 10.

VA = [ p1 (12 bits) | p2 (10 bits) | offset (10 bits) ] p1 → outer page table (2¹² = 4096 entries) p2 → inner page table (2¹⁰ = 1024 entries/table) Known as "forward-mapped page table"
64-bit — why even three levels isn't always enough▼

64-bit VA + 4KB pages → 2⁵² page table entries in flat table.

Two-level: outer table = 2⁴² entries = 16 TB. Still too large.

Modern x86-64 uses 4-level page tables. Each level = 1 extra memory access. TLB is essential to avoid 4+ accesses per logical address.

Topic 08

Deadlock

Definition: A set of processes permanently blocked — each holding resources needed by another. No process can proceed. All 4 Coffman conditions must hold simultaneously.
Coffman's 4 Conditions
1. Mutual Exclusion▼
At least one resource is non-shareable. Only one process can use it at a time. Others must wait.
2. Hold and Wait▼
A process is holding ≥1 resource AND waiting for additional resources held by other processes.
3. No Preemption▼
Resources cannot be forcibly taken. Released only voluntarily after the process completes its task.
4. Circular Wait▼
Cycle: P0 waits for P1's resource, P1 waits for P2's, ..., Pn waits for P0's resource.
RAG & Wait-For Graph

RAG: Processes=circles, Resources=squares. Request edge: Ti→Rj. Assignment edge: Rj→Ti.

RAG StateConclusion
No cycleNo deadlock
Cycle + single instance resourcesDeadlock!
Cycle + multi-instance resourcesPossible — verify

WFG: Remove resource nodes from RAG. Pi→Pj means Pi waits for resource held by Pj. Cycle = deadlock. Only valid for single-instance resources.

Banker's Algorithm
Need = Max − Allocation

Safe if: ∃ sequence where all
processes can finish

Safety Algorithm:

Work = Available. Finish[i] = false for all i

Find i: Finish[i]=false AND Need[i] ≤ Work

Work += Allocation[i]; Finish[i] = true. Repeat from 2.

If all Finish[i]=true → SAFE, else → UNSAFE

Example: Available={1,1,1}

PAlloc A,B,CMax A,B,CNeed A,B,C
P01,0,12,1,11,1,0
P10,1,01,2,11,1,1
P21,1,01,1,20,0,2
P30,0,11,1,11,1,0
